User Testing

Cognitive Walkthrough

A cognitive walkthrough it an evaluation method in which evaluators work through a series of tasks and ask a set of questions from the perspective of the user. Cognitive walkthroughs are important because they allow low cost usability testing to take place with only a prototype and they test the usability of basic functions from the user’s point of view. The challenge for me was getting my evaluators to vocalize their action and provide more feedback. I would ask them follow-up questions to elicit more information. Lastly, the tasks related to the website were very basic and quick which hindered the amount of information I received. View the Cognitive Walkthrough.

User Stories

User stories provide a human factor to otherwise impersonal documentation. They all for a conversation to take place on requirements, features, and functionality. I chose my user stories based on what hiring managers and people in the position I am applying for would look for. These are core elements of my portfolio website yet when describing them in the user stories I was able to get more insight and how the user thinks compared to what I think. View the User Stories.

Summary of Findings

As far as preliminary planning and research for the site, I found competitor analysis to be very valuable. I was able to grasp the current trends and benchmarks of sleek modern portfolio websites. Many of my design inspirations stemmed from my competitors design choices. This culminated to finalizing features and functionality that I thought would add value to my site and that were evident among my competitors. The feature value matrix for me was more of ranking the importance of what features were absolutely necessary and others that were secondary. Having that resource was valuable especially when I had to pull back on features due to the lack of time. The user interviews weren’t extremely valuable to me. I found that I was trying to reinforce my ideas rather than gather new insight. My own research proved to be much more valuable.


When conducting User Testing it was challenging to extract solid feedback from the users I was testing. It may have been my lack of experience on my part or a poor selection of user participants. Either way, I was not able to gather enough information to make concrete improvements. Many people commented about the monochrome design colors. The feedback was both good and bad. Granted I find the color scheme to best represent my personality and content. As stated above my cognitive walkthroughs were based around very simple tasks. For this reason I was not able to extract any real feedback and it mostly reinforced my decision to place things in certain areas. Lastly, I had trouble with user stories. One issue being that I didn'nt implement all the features I wanted to. The second being I had trouble puttin myself in a users shoes after being involved with the project for so long.

While I was implementing the site I found time and my technical ability to be a limiting constraint. I was initially very ambitious with what I wanted to accomplish. I did not schedule my time efficiently or break down the work into increments. I had to scrap many of the features that were low on the priority list. This was upsetting as I wanted my portfolio to have fancy features. I focused more on the content, as suggested by the professor, and tried to present in a way that was informative and intuitive.
